    Object lifetimes in Rezolver

    Tip

    Read Creating and using a container if you haven't already done so.

    As with any IOC container, Rezolver understands the concept of object lifetimes, which are implemented as different ITarget implementations that can be registered in an ITargetContainer.

    What is a lifetime?

    In a crude sense, an object lifetime describes when a new object is created and whether a previously created object is subsequently returned from a container's Resolve method.

    Note

    In practise, it's not that simple - because an IOC container's understanding of a lifetime does not directly map to whether a new instance is actually created at all.

    In IOC world - it's more accurate to say that the lifetime determines when the action that's been associated with a given service type is actually executed.

    Rezolver understands three primary lifetimes, which should be familiar to anyone who's used an IOC container before:

    Transient

    In general programming terms, a transient object is simply one which is created when needed; and thrown away when it's fulfilled its purpose.

    This typically translates to calling the new operator with the constructor of a particular type, and then allowing that object to go out of scope, as in this snippet:

    // Factory method always creates a new instance
static MyObject CreateObject()
{
    return new MyObject();
}

static MyObject _transient = CreateObject();

void Test()
{
    var transient1 = CreateObject();
    var transient2 = CreateObject();
    //note - using XUnit nomenclature here
    Assert.NotSame(transient1, transient2);
    Assert.NotSame(_transient, transient1);
    Assert.NotSame(_transient, transient2);
}

    This is what most developers think of when they talk about a transient object, and it might well be correct, but it's not the whole story.

    If transiency is defined as the caller **knowing** that new is called (or some other method by which we can invoke a constructor - such as via Activator.CreateInstance) - then how do we define the lifetime of MyObject when we define CreateObject as follows:

    private static IMyObjectService _service = new MyObjectService();

static MyObject CreateObject()
{
    return _service.GetObject();
}

    The answer is: we don't know what the **object's* lifetime* is, because we no longer know how the object is produced.

    But we do know that we call the service's GetObject() method every time we want an object - so therefore we treat it as transient.

    In

    And this is where Rezolver's concept of transience comes into play, and it's derived from the role of the ITarget interface in our standard container types.

    Tip

    We don't have transient 'objects' in Rezolver, we have targets which describe transient actions that the concept of transiency in Rezolver (or in any IOC container) is less connected with the actual instance that's returned from the Resolve method, but more the action that's performed to produce that object.

    In the case of Rezolver, most of the target types in the Rezolver.Targets namespace will yield objects which are implicitly transient, because they don't impose their own lifetime rules.
